## Directory Purposes

**src/:**
- Purpose: Source TypeScript files
- Contains: Entry point, API client, tool definitions
- Key files: `index.ts` (transport logic), `sonix-client.ts` (API abstraction), `tools.ts` (tool handlers)

**dist/:**
- Purpose: Compiled JavaScript output
- Contains: `.js` and `.d.ts` files mirroring src/ structure
- Generated: Yes (by `npm run build`)
- Committed: No (in .gitignore)

**.planning/codebase/:**
- Purpose: GSD (automated planning) analysis documents
- Contains: ARCHITECTURE.md, STRUCTURE.md, CONVENTIONS.md, TESTING.md, STACK.md, INTEGRATIONS.md, CONCERNS.md
- Generated: Yes (by /gsd:map-codebase commands)
- Committed: Yes

## Key File Locations

**Entry Points:**
- `src/index.ts`: Main server entry point with dual-transport support (stdio + HTTP); exports no public interface, runs on require
- `dist/index.js`: Compiled executable; bin target in package.json points here
- Dockerfile: Builds multi-stage (Node 22.15 slim), compiles TypeScript, runs dist/index.js

**Configuration:**
- `package.json`: Project metadata, dependencies (@modelcontextprotocol/sdk, express, zod), build scripts
- `tsconfig.json`: Compilation to ES2022, module: NodeNext, strict mode enabled
- `.env.example`: Template for SONIX_API_KEY environment variable
- `.mcp.json`: Local MCP server configuration (hardcoded path for testing)
- `Dockerfile`: Production build target, HTTP mode default (TRANSPORT=http), port 3000

**Core Logic:**
- `src/sonix-client.ts`: HTTP client for Sonix.ai API, 15 public methods (list, get, upload, delete, update, summarize, translate, etc.)
- `src/tools.ts`: MCP tool registration, 13 tools defined (list_media, get_media, upload_media, get_transcript, search_media, delete_media, update_media, summarize_media, list_folders, create_folder, translate_transcript, get_translation)
- `src/index.ts`: MCP server factory, transport selection, session management, rate limiting, CORS

**Testing:**
- No test files present; see TESTING.md for testing approach

## Naming Conventions

**Files:**
- TypeScript source: camelCase (`sonix-client.ts`, not `SonixClient.ts`)
- Generated: dist/ mirrors src/ structure exactly
- Config: lowercase with hyphens (`.mcp.json`, `.env.example`, `package.json`)

**Directories:**
- lowercase, plural for collections (src/, dist/, .planning/)
- dot-prefix for hidden config (.git/, .planning/)

**TypeScript Classes:**
- PascalCase: `SonixClient`, `McpServer` (from SDK)
- Exported from specific modules: `SonixClient` from `sonix-client.ts`

**Functions:**
- camelCase: `createServer()`, `startHttp()`, `startStdio()`, `registerTools()`
- Private functions in modules: `log()`, `sessionHash()`, `extractApiKey()`, `verifySession()`
- Arrow functions for tool handlers: `async ({ param1, param2 }) => { ... }`

**Variables:**
- camelCase: `apiKey`, `sessionId`, `sessionIdGenerator`, `corsOrigin`, `globalLimiter`
- SCREAMING_SNAKE_CASE for constants: `BASE_URL`, `REQUEST_TIMEOUT_MS`, `MAX_RESPONSE_BYTES`, `ACCEPT_HEADERS`, `SESSION_TTL_MS`, `MAX_SESSIONS`
- Map/collection instances: `sessions` (Map<string, Session>)

**Types:**
- Interfaces: `Session` (src/index.ts lines 65-70), `ToolResult` (src/tools.ts lines 6-9)
- Unions: Response format types (implicit in handler return values)
- Zod schemas: inline in tool definitions, named after parameter groups (e.g., `z.enum([...])`)

## Where to Add New Code

**New Tool:**
- Primary code: Add `server.tool(...)` call in `src/tools.ts` within `registerTools()` function
- Client method: If tool calls new Sonix API endpoint, add method to `SonixClient` class in `src/sonix-client.ts`
- Pattern: Follow existing tool structure (Zod schema, error handling with try/catch, ToolResult return)

**New API Endpoint:**
- Client method: Add to `SonixClient` class in `src/sonix-client.ts`
- Pattern: Use private `request()` helper with proper headers/method/body; call `readBody()` for large responses
- Tool handler: Create new tool in `src/tools.ts` that calls the new method

**HTTP Middleware:**
- Location: `src/index.ts`, inside `startHttp()` function, after Express setup (lines 102-115 show existing middleware)
- Use: `app.use()` for global middleware before route definitions (line 135 and onward)
- Pattern: Follow rate limiter setup (lines 118-134)

**New Configuration Option:**
- Environment variable: Add to `.env.example` with description
- Usage: Read in `startHttp()` or `startStdio()` via `process.env.VARNAME`
- Example: `CORS_ORIGIN` (line 105), `PORT` (line 96), `TRANSPORT` (line 270)

## Build & Deployment

**Local Development:**
```bash
npm install          # Install dependencies
npm run build        # Compile TypeScript to dist/
npm run watch        # Watch mode (recompile on changes)
npm start            # Run dist/index.js (stdio mode)
TRANSPORT=http PORT=3000 npm start  # HTTP mode
```

**Docker:**
- Multi-stage build: Node 22.15 slim builder → runner
- Stage 1 (builder): npm ci, tsc compile
- Stage 2 (runtime): npm ci --omit=dev, copy compiled dist/, expose port 3000
- Default: HTTP mode (TRANSPORT=http), requires SONIX_API_KEY env var

**Package Structure:**
- "type": "module" — ES6 modules (import/export syntax)
- "bin": mcp-server-sonix points to dist/index.js
- "files": dist/ only (excludes src/, tests, docs from npm package)
